# Бриф: Telegram Mini App — каталог бьюти-мастера

> Версия: MVP (v1.0)
> Дата: 2 марта 2026

---

## Общая логика

Каждый мастер подключает **своего** Telegram-бота. Один Mini App, но у каждого мастера — отдельная "дверь" с личным именем.

```
@anna_beauty_bot    →  Бот мастера Анны (её клиенты пишут сюда)
@olga_nails_bot     →  Бот мастера Ольги (её клиенты пишут сюда)
@maria_brows_bot    →  Бот мастера Марии (её клиенты пишут сюда)
```

Все боты подключены к одному бэкенду. Mini App (фронтенд) — один и тот же, но по `bot_id` из `initData` определяет, какого мастера показать.

**Определение режима (мастер vs клиент):**
- `initData.user.id` совпадает с `masters.telegram_id` этого бота → **админка мастера**
- Не совпадает → **каталог для клиента**

Мастер не видит чужих клиентов. Клиенты не знают о существовании других ботов. Полная изоляция.

### Экран М0: Подключение бота — до первого входа в Mini App

Мастер подключает бота через лендинг (веб-страница) или через платформенного бота-помощника.

#### Процесс подключения:

```
1. Мастер открывает лендинг или платформенного бота
2. Видит инструкцию: "Создайте бота за 2 минуты"
3. Пошаговая инструкция с картинками:
   а) Откройте @BotFather в Telegram
   б) Отправьте /newbot
   в) Придумайте имя (например: "Мастер Анна — запись")
```

**Что система делает автоматически (за 2-3 секунды):**

| Действие | Telegram API метод |
|----------|-------------------|

## Часть 7. Telegram-бот (мультибот-архитектура)

Один сервер обрабатывает webhook'и от **всех** ботов мастеров. Python + aiogram + aiohttp.

### Архитектура:

```
@anna_beauty_bot   → webhook → POST /webhook/bot123456   ─┐
@olga_nails_bot    → webhook → POST /webhook/bot789012   ─┼→  Один сервер (aiohttp)
@maria_brows_bot   → webhook → POST /webhook/bot345678   ─┘         │
                                                                      ▼
                                                              По bot_id из URL
                                                              находим мастера в БД
                                                                      │
                                                                      ▼
                                                              Отправляем ответ через
                                                              токен этого бота
```

**Как работает:**
1. При подключении нового бота — `setWebhook(url=f"{SERVER_URL}/webhook/bot{bot_id}")`
2. Сервер получает update на `/webhook/bot{bot_id}`
3. По `bot_id` из URL находит мастера и его токен в БД
4. Создаёт экземпляр Bot(token) для ответа (кэшируется в памяти)
5. Обрабатывает команду / отправляет уведомление

**Команды каждого бота:**
- `/start` — приветствие + кнопка "Открыть каталог" (inline с Mini App)
- `/booking` — "Мои записи" (inline кнопка → Mini App)
- `/help` — контакты мастера

## Часть 7.1. Безопасность

### Хранение токенов ботов
- `bot_token` — чувствительные данные (эквивалент пароля к боту)
- Хранить зашифрованными в БД (AES-256, ключ в переменных окружения)
- Никогда не отдавать на фронтенд
- Использовать только на бэкенде (бот-сервер и Edge Functions)
- При удалении аккаунта мастера — удалить webhook, очистить токен

### Проверка initData
- Каждый запрос от Mini App содержит `initData`
- Бэкенд проверяет подпись через HMAC-SHA-256 (ключ = bot_token конкретного бота)
- Без проверки — любой может подделать запрос от имени другого пользователя
- В мультибот-архитектуре: Mini App передаёт `bot_id` → бэкенд находит токен → проверяет подпись

### Изоляция данных (RLS)
- Мастер видит только свои данные
- Клиент видит только каталог мастера, чей бот открыл
- Записи видны только участникам (мастер + клиент)
